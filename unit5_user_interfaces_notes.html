1

Introduction

This chapter discusses user interfaces for communication between human information seekers and information retrieval (IR) systems.

Information seeking is an imprecise process — users often have only a fuzzy idea of what they want when they start.

What should a good UI do?

  • Help users understand and express their information needs
  • Help formulate queries effectively
  • Help users choose among available information sources
  • Help understand search results
  • Help track progress of the search
📌 Key Point
The human-computer interface is less well understood than other IR aspects because humans are more complex — their motivations and behaviours are harder to measure. This area is also changing rapidly.
2

Human-Computer Interaction

An effective human-computer interface almost disappears — users feel in control, competent, and clear. When well-designed, the interface lets users concentrate on their actual goal.

2.1 Design Principles

Shneiderman's key principles for information access UI design:

1
Offer Informative Feedback Give users constant feedback about: the relationship between their query and retrieved documents, relationships among documents, and relationships between documents and collection metadata. Letting users control when/how feedback is given ensures an internal locus of control.
2
Permit Easy Reversal of Actions Users should be able to undo or backtrack without losing their context or earlier work.
3
Reduce Working Memory Load Information access is iterative — goals shift as new info is found. Good UIs help by: keeping track of choices made during a search, letting users return to abandoned strategies, retaining context across sessions, and providing browsable suggestions (related terms, topic lists, source lists).
4
Provide Alternative Interfaces for Novice and Expert Users Simple interfaces are easy to learn but less flexible. Powerful interfaces give experts more control. A common fix is scaffolding — a simple interface for beginners with advanced options available for experts. The key: intuitive bridges between simple and advanced modes.
💡 Simplicity vs. Power Tradeoff
A major design question is how much information to show about the system itself. New users may not know how to handle complex features. Experienced users can benefit from seeing suggested terms, weights, etc. Finding the right balance is a core design challenge.
2.2 The Role of Visualization

Information Visualization attempts to provide visual depictions of very large information spaces. Humans are highly attuned to images — a well-designed visual can communicate information far faster than any other method.

Key visualization concepts:

ConceptMeaning
Brushing & LinkingConnecting two or more views of the same data — a change in one view updates all other views automatically.
Panning & ZoomingLike a movie camera — panning scans sideways, zooming moves in for detail or back for a wider view.
Focus-plus-ContextOne part of the view (the focus) is shown in detail while surrounding content is compressed. Partly solves the problem of losing context when zooming in.
⚠ Challenge
Visualizing abstract or textual information is especially hard because language communicates abstract ideas with no obvious physical form.
2.3 Evaluating Interactive Systems
  • People differ in spatial ability, memory, reasoning, verbal aptitude, age, and cultural background — all affecting UI usability.
  • Precision and recall (standard IR metrics) are less appropriate for evaluating interactive systems.
  • Better metrics for interactive systems: time to learn, time to complete tasks, error rates, retention of interface over time.
  • Empirical data from user studies is expensive and hard to generalise.
  • Nielsen advocates heuristic evaluation: an informal assessment of UI affordances against general usability properties — without requiring statistically significant results.
3

The Information Access Process

A person engaged in information-seeking has goals and uses a search system as a tool to achieve them. Goals range widely — from finding a plumber to writing a scholarly article.

A study of business analysts found three main types of information-seeking behaviours:

1
Monitoring Tracking a familiar topic over time (e.g., researching a competitor each quarter).
2
Planned Searching Following a structured/routine set of searches to achieve a known objective (e.g., keeping up with business best practices).
3
Exploratory Searching Open-ended exploration to become familiar with a new subject or industry.
4. Models of Interaction

The standard interaction cycle for information access:

  1. Start with an information need.
  2. Select a system and collection to search.
  3. Formulate a query.
  4. Send the query to the system.
  5. Receive results (information items).
  6. Scan, evaluate, and interpret the results.
  7. Either stop, or…
  8. Reformulate the query and repeat from Step 4.
❌ Problems with the Standard Model
  • Assumes the user's information need stays the same throughout.
  • Assumes the goal is to find a "perfect" set of results.
  • Ignores dynamic learning — users often change goals mid-search.
  • Presents users with long, unstructured result lists.
  • Doesn't account for hyperlink navigation, browsing, or topic scanning.
The Berry-Picking Model (Marcia Bates)

This model challenges the traditional view by recognising how people actually search in real situations.

  • Information needs evolve — as users find new info, their goals may shift or branch.
  • Users gather bits and pieces from different sources (like picking berries one by one).
  • There is no final "perfect" result set — the process is incremental and adaptive.
  • Emphasises learning, scanning, exploring, and changing strategies.
✅ Example
A user searching for "history of climate change" might read an article, discover a new interest in "climate policy in the 1980s", shift focus, search again, and gradually build knowledge from many different documents.

Implications for User Interfaces: To support real-world information seeking, interfaces should let users:

  • Adjust goals mid-search
  • Switch strategies easily
  • Track progress and store intermediate results
  • Resume previous search paths
  • Integrate multiple strategies (e.g., browsing and querying together)
  • Support exploration, navigation, and serendipitous discovery
Modes of Interacting with Information
ModeDescription
ScanningQuickly looking over structures — titles, labels, links.
QueryingActively searching by inputting keywords or selecting terms.
NavigatingClicking through links to reach related content.
BrowsingCasual, less goal-directed exploration of content.
💡 Remember
These modes often interact — e.g., scanning leads to browsing, which leads to querying. The output of one action (e.g., finding a useful term) should naturally feed into the next (e.g., forming a new query).
5

Non-Search Parts of the Information Access Process

Information seeking is only one part of the full work process. Between searching sessions, users do: reading, annotating, and analysis.

O'Day and Jeffries found that 80% of analysis work falls into six types:

  • Finding trends
  • Making comparisons
  • Aggregating information
  • Identifying a critical subset
  • Assessing
  • Interpreting

The remaining 20%: cross-referencing, summarising, finding visualisations for reports, and miscellaneous.

🔑 Key Idea — Sensemaking (Russell et al.)
Most effort in information work is in the synthesis of a good representation — formulating and crystallising the important concepts. User interfaces should allow search/retrieval AND analysis/synthesis to be tightly interwoven.
6

Earlier Interface Studies

❌ Earlier Systems (Pre-Internet Era)

  • Used only bibliographic records (titles, abstracts).
  • Required professional intermediaries to search.
  • Used Boolean logic (AND, OR, NOT) — rigid and hard to use.
  • Text-only interfaces (TTYs) required memorising commands.
  • Switching databases was slow and complex.

✅ Modern Systems

  • Full-text search (actual content of web pages/papers).
  • End users (non-experts) do the searching themselves.
  • Statistical ranking (like Google's relevance ranking).
  • Documents are hyperlinked — seamless navigation.
  • Graphical interfaces with direct manipulation.
  • A "near miss" result is still useful — follow links to better content.
1. Starting Points

Users usually start with very short queries, inspect results, and modify incrementally. An empty form or blank screen gives no clues to help users start.

Four main types of starting points:

  • Lists — lists of sources/collections to choose from
  • Overviews — summaries of what's in the collection
  • Examples — sample documents to orient the user
  • Automated source selection — system picks the best source
7

Lists of Collections

Online systems like LEXIS-NEXIS start by showing a long list of source names and asking users to guess which ones are relevant — often with little additional information about each source.

  • Experienced users learn which sources suit their needs through trial-and-error, training, or recommendations.
  • They use "favorites" lists (also called bookmarks/hotlists on the Web).
  • Challenge: For new or unfamiliar topics, favorites don't help. Traditional interfaces make discovering new sources difficult for regular users.
8

Overviews

Overviews show the topic domains represented in a collection, helping users select or eliminate sources. Shneiderman's model: overview → zoom/filter → details.

Three types of overviews:

1
Topical Hierarchies (Manual Categories) Tree structures showing categories and subcategories. Documents organised by predefined themes. Users browse through categories to find relevant content.
2
Automatically Generated Overviews (Clustering) Machine learning groups documents by content similarity without human input. Helps reveal hidden patterns or groupings in large datasets.
3
Link-Based Overviews (Co-Citation / Network Analysis) Based on connections (citations, links, shared references). Documents frequently cited together may be thematically related. Relationship networks define the groupings.
Category / Directory Overviews

Examples of category-based systems: Traditional Libraries (subject headings), MEDLINE/MeSH (~18,000 categories), ACM (~1,200 keyword categories), Yahoo! Directory.

Problems with Category Systems:

  • Poor Interfaces: Users don't use subject headings because the UI is bad (e.g., old command-line).
  • Hard to Navigate: Users must guess the right category; wrong guesses require backtracking.
  • Deep navigation loses context — users forget what they were originally looking for.
  • Limited search capability: Most systems don't allow searching across multiple categories simultaneously.
  • Large screen space needed to display full hierarchies.

Systems attempting to improve on these problems:

  • Internet Grateful Med: Web interface to MeSH — but clicking a category loses search context and makes navigation clumsy.
  • MeSHBrowse: Interactive browsing of MeSH hierarchy in 2D tree view — limited to browsing, no direct document search.
  • HiBrowse: Faceted navigation (Disease, Therapy, Age Group, etc.), shows document counts dynamically — but querying is not tightly integrated.
Automatically Derived Collection Overviews (Clustering)

Unsupervised clustering groups documents by similarity. Each cluster has a centroid describing its main theme. This lets users explore a high-level overview without reading every document.

Scatter/Gather System: Clusters a large collection → shows summary terms + sample titles → user selects clusters → re-clusters for deeper exploration.

⚠ Limitations
  • Scatter/Gather alone was less effective than traditional search for finding relevant documents.
  • When combined with search (clustering query results), performance improved significantly.
  • No single clustering algorithm consistently outperforms others.
  • Generated themes can be inconsistently specific/broad, confusing users.

Visual representations of clustering:

  • Scatter-plot layouts: Each document = a small icon in 2D space. Examples: BEAD, Galaxy of News, ThemeScape (3D topographic "mountains").
  • Kohonen Feature Maps: A neural network arranging documents in 2D grids by theme. Adjacent regions are semantically related.
  • Inter-document similarity graphs: Networks showing how documents relate to each other.

Evaluations of Graphical Overviews

User studies show graphical overviews are often not very effective, especially for general users.

📊 Key Study: Kohonen Map vs Yahoo!
  • 11/15 users on Kohonen map found an interesting page, but only 2 could later find it again.
  • 14/16 users on Yahoo! succeeded, but only 2 could navigate back on the Kohonen map.
  • Conclusion: Yahoo!'s simple text-label view was more effective and navigable.
  • Users liked jumping easily between map areas; disliked lack of hierarchy and confusing labels.
Co-Citation Clustering for Overviews

If two documents are cited together by another document, they are co-cited. Co-cited documents likely share related themes. These relationships build a similarity matrix that is then clustered.

  • Helps identify thematic clusters and central/key authors.
  • Referral Web project: applies co-citation logic to web documents.
  • Authority Pages (Kleinberg): identifies good starting points for topics — pages that many others link to.
9

Automated Source Selection

Selecting the right source (database, website, search engine) is critical but hard to do manually. Automated methods help match user needs to the best sources.

Two main approaches:

🤖 Modeling User + Source (AI-driven)

  • Build a model of the user's information need.
  • Build a model of each source's content.
  • Match them to find the best source.
  • Used in: user modeling systems, intelligent tutoring, online help.
  • Challenge: Complex and resource-heavy.

📊 Content-Based Matching (Practical)

  • Uses textual representations of sources, directly compared to the query.
  • GIOSS: checks how often query terms appear in databases; selects the best one.
  • SavvySearch: uses both query and user click-through behaviour to rank sources.
🔍 Alternative: Metasearch & Collection Fusion
Instead of picking one best source, send the query to multiple sources at once and combine results. This is what metasearch engines do (e.g., Dogpile, Metacrawler). Collection fusion — how to combine heterogeneous results meaningfully — is a key research area.
10

Query Specification

To formulate a query, a user must select collections and specify terms. Five interaction styles exist: command language, form filling, menu selection, direct manipulation, and natural language.

10. Boolean Queries
❌ Problems with Boolean Queries
  • Most users find the syntax counter-intuitive — many misinterpret AND/OR.
  • English AND ("dogs and cats") ≠ Boolean AND (documents about BOTH dogs AND cats together).
  • No ranking — documents either match or they don't. No measure of "how well" they match.
  • Complex syntax (parentheses, operator precedence) overwhelms casual users.

Web solutions: allow users to choose "all words" (AND) or "any words" (OR) without explicit Boolean operators. The '+' prefix (Altavista) was widely adopted but often misunderstood.

11. From Command Lines to Forms and Menus

TTY command-line interfaces required users to type cryptic commands and remember syntax (e.g., FIND PA darwin AND TW species).

The Web-based Melvyl system improved on this with form filling and menu selection: users select metadata types from listboxes and don't need to remember attribute names. It also retains context between searches.

12. Faceted Queries

A faceted query divides the user's query into topics/facets, each of which should be present in retrieved documents. This approach helps when pure Boolean queries return too many or zero results.

✅ Example
A query on "drugs for prevention of osteoporosis" is divided into facets:
(osteoporosis OR "bone loss") AND (drugs OR pharmaceuticals) AND (prevention OR cure)

Quorum Ranking: ranks documents by how many facets they satisfy. A document with hits on 3 out of 4 facets is ranked higher than one with 30 hits on just 1 facet.

13. Graphical Approaches to Query Specification

Direct manipulation interfaces provide alternatives to command-line syntax. Properties: (1) continuous representation of the object of interest, (2) physical actions instead of complex syntax, (3) rapid, reversible operations with immediate visible impact.

  • Venn Diagram / VQuery: Overlapping ovals represent ANDs; selecting outside overlap = OR; unselected oval in active area = NOT. Limited to ~3 terms in standard Venn format.
  • Filter-Flow Model (Young & Shneiderman): User selects attribute values; placing attributes in sequence = AND (conjunction); in parallel = OR (disjunction). Width of "water flow" shows how many documents pass through.
  • Block-Oriented Diagram (Anick et al.): Natural language query broken into blocks in rows/columns. Same row = AND; same column = OR. Users activate/deactivate blocks to rapidly experiment with combinations.
  • Magic Lenses (Fishkin & Stone): Transparent lenses act as filters on a 2D document space. Stacking lenses combines conditions as a conjunction.

14. Phrases and Proximity

Proximity information significantly improves precision. Different methods for specifying phrases:

  • LEXIS-NEXIS: white w/3 house = "white" within 3 words of "house".
  • Web search engines: Enclose terms in quotation marks for exact phrase match.
  • Limitation: Exact phrase matching misses near-matches (e.g., "President Abraham Lincoln" won't be caught by "president lincoln").
  • Best approach: Allow exact phrases but treat them as small proximity ranges with exponential fall-off in weight.
15. Natural Language and Free Text Queries
  • Statistical ranking allows users to query naturally without Boolean operators.
  • Drawback: Users get little feedback about why a document ranked the way it did.
  • Mandatory terms: Some systems let users mark certain terms as required within a natural language query.
  • NLP parsing: Some systems parse the query's syntax to extract concepts (e.g., Murax extracts whether the user is asking for a person, place, or date).
  • Question Answering systems: Ask Jeeves maps user queries to question templates linked to curated web pages. FAQ Finder matches queries to FAQ question-answer pairs.
16

Document Surrogates

A document surrogate is a brief representation of a document — typically title, source, date, length, and a relevance score — used to display results lists without showing the full document.

  • Some systems offer short view (just title + metadata) and detailed view (abstract or excerpt).
  • Clicking the title brings up the full document.
  • Web search engines generate excerpts automatically (usually first few lines of the page).

Query Term Hits Within Document Content

Highlighting query term occurrences within retrieved documents helps users quickly find the most relevant passages. Color highlighting and auto-scrolling to the first match are common techniques found to be consistently useful.

KWIC (Keyword-In-Context)

Extracts sentence fragments containing the query terms and shows them as part of the document surrogate. Unlike an abstract (which summarises the document), KWIC shows how the query terms are used in context.

  • Shows which subsets of query terms occur and in what context.
  • Best fragments: near the beginning of the document AND containing the largest subset of query terms.
  • Not commonly shown in web search (engines don't usually store position information).

SuperBook: Context via Table of Contents

SuperBook uses a large document's Table of Contents (TOC) to display query results in context. The TOC is shown on the left; full text is on the right. A focus-plus-context mechanism dynamically expands the current section while compressing others.

  • After a query, sections with hits are made larger; others are compressed.
  • Query terms in chapter/section names are highlighted in reverse video.
  • User studies found SuperBook users were faster and more accurate than users of paper manuals or standard online systems.
  • Query term highlighting was found to be at least partially responsible for improvements.

Categories for Results Set Context

Category metadata can also help place query results in context (beyond just a TOC):

  • CORE project: Extended SuperBook to 1000+ full-text chemistry articles, using a chemistry category hierarchy instead of a TOC.
  • DynaCat system: Organises retrieved documents using pre-selected categories from MeSH, based on the query type (e.g., "Treatment-Adverse Effects").
  • SONIA system: Combines unsupervised clustering + user adjustments + supervised categorisation for personalised organisation.

Using Relevance Judgements

Relevance feedback: the user selects documents that appear relevant; the system uses features from these to revise the query. The new query is re-executed. This is a proven effective technique for improving retrieval.

18. Interfaces for Standard Relevance Feedback
  • Most common: a results list with checkboxes beside each title for marking relevant documents.
  • After marking, the system either automatically reweights the query, or generates a list of suggested terms for the user to select.
  • Web engines use "More like this" as a simpler one-click version.
  • Incremental relevance feedback (Aalbersberg): works well given only one relevant document at a time.
📄 Refer to Figure 10.25 (Standard relevance feedback interface) in the original PDF.
19. Studies of User Interaction with Relevance Feedback

Koenemann and Belkin tested four conditions for user control during relevance feedback:

ConditionDescriptionOutcome
ControlNo relevance feedback — manual query reformulation only.Baseline performance
OpaqueUser selects relevant docs; sees revised rankings.Better than control
TransparentUser sees which terms were added by the system.Between control and penetrable
PenetrableUser chooses which suggested terms to add before re-execution.Best performance — fewest iterations (avg 5.8 vs 8.2 in control)
✅ Key Finding
All conditions with relevance feedback outperformed the no-feedback control. Users with the most control (penetrable) achieved better results in fewer iterations.
20. Fetching Relevant Information in the Background

Semi-automated assistants / recommender agents monitor users' behavior over time to predict future needs.

  • Kozierok & Maes system: Predicts how users handle email and meetings by learning from past actions. Uses nearest-neighbours to suggest actions.
  • Syskill and Webert: Learns a user profile from explicit relevance judgements of web pages browsed. Annotates links on new pages.
  • Letizia: Monitors browsing behaviour and concurrently evaluates reachable links to suggest next moves. Uses implicit feedback only.
  • Butterfly: Prefetches citation links from the paper the user is currently reading. Shows incrementally growing citation list with animation.
21. Group Relevance Judgements

Social recommendation / collaborative filtering: Uses relevance judgements from many different users to rate information. Highly effective for rating items where taste matters (movies, music). More recent work combines group judgements with content information.

22. Pseudo-Relevance Feedback

The system simply assumes its top-ranked documents are relevant and uses them to augment the query — no user input required. Highly effective when the original query is long and precise.

Interface Support for the Search Process

1. Interfaces for String Matching

The simplest search need: the "Find" operation — searches within the currently viewed document. Usually a simple string match (no ranking). Key issue: the result list should be cleared as soon as a new query is typed, to avoid misleading the user.

2. Window Management

Interface designers must decide how to lay out information within the interface:

Layout TypeAdvantagesDisadvantages
MonolithicDesigner controls layout; all info simultaneously visible; familiar positionsWorks best fullscreen; limits number of views; hampers integration with other apps
Tiled WindowsAll windows non-overlapping and visibleFixed; less flexible arrangement
Overlapping WindowsFlexible; can show more than fits on screenCan quickly become crowded and disorganized; context switching is hard

Workspace concept (Card et al.): grouping functionally related windows together matches users' goal structure better than individual windows.

4. Examples of Poor Use of Overlapping Windows
❌ Bad Practice Example (LEXIS-NEXIS)
The legacy LEXIS-NEXIS interface doesn't use window context when switching functions — viewing metadata causes the results list to disappear entirely instead of using a pop-up. Metadata is rendered in poorly formatted ASCII even though a graphical interface is available. Full text is shown in a tiny, non-expandable area.
5. Retaining Search History

Good UIs should show: available choices at the current point, prior moves made, short-term tactics, longer-term strategies, and allow annotations of choices and findings.

  • Simple history: a list of commands executed earlier.
  • Graphical history: tracks commands and results together.
  • VISAGE system: Saves search states as miniatures in a "slide sorter" view — visual, easy to recognise.
📄 Refer to Figure 10.31 (VISAGE interaction history visualization) in the original PDF.
6. Integrating Scanning, Selection, and Querying
⚠ Current Problem
Most UIs don't support strategies or sequences of moves well. Even taking results from one query and using them as input to a later query is not well supported. Scanning, selection, and querying need to be better integrated.

Exceptions:

  • Web-based Melvyl: allows modifying prior queries and re-executing them.
  • DLITE / Rooms: support workspace-based storage and reuse of prior state.
  • Cat-a-Cone: tightly integrates category browsing and querying (see next section).
22

Cat-a-Cone: Integrating Category and Text Scanning with Search

Cat-a-Cone integrates querying and browsing of very large category hierarchies with their associated text collections. Uses 3D + animation from the Information Visualizer.

Key design choices:

  • Separation of category hierarchy representation from document representation — allows fluid interaction between browsing and search.
  • Retrieved documents stored in a 3D book representation — user "riffles" pages to see corresponding changes in the category hierarchy.
  • Shows which parts of the category space reflect each retrieved document's semantics.
  • Supports relevance feedback by suggesting additional related categories.

Starting points:

  • Type a category name → see which parts of the hierarchy match it.
  • Type a free text query → retrieve documents → open them in the book → see associated categories in hierarchy.

Trends and Research Issues

Human-computer interaction is receiving increasing recognition in computer science. This chapter has shown how rapidly the field is evolving and how many open research questions remain.

Key themes and open problems:

  • Scanning, selection, and querying need to be better integrated in search interfaces.
  • Supporting multi-session strategies — carrying results from one search session into another — remains largely unsolved.
  • Interfaces need to support the full range of information access activities (search, analysis, synthesis) in an integrated way.
  • Workspace and history mechanisms need further development and integration with general search.
  • Flexibility is needed, but within constraints — unrestricted interaction can lead to erroneous or wasteful behaviour.
  • Visual representations (maps, clusters) show promise but are generally less effective than text-based representations for direct search tasks.
  • Cat-a-Cone demonstrates some key design decisions that address these challenges — hierarchical category display, first-class document representation, alphabetical searching, and relevance feedback through category suggestions.
💡 Summary of Key Design Lessons
  • Provide informative feedback at every stage of the search process.
  • Support both novice and expert users with scaffolded interfaces.
  • Reduce memory load through history, context retention, and browsable suggestions.
  • Integrate scanning, browsing, querying fluidly — output of one should feed the next.
  • Graphical tools like TileBars and InfoCrystal can help users visualise complex result sets.
  • Relevance feedback — especially when users have control (penetrable mode) — significantly improves results.
  • Workspace/history mechanisms help users maintain context across multi-session searches.
