Introduction

As document collections grow, the cost of managing them with traditional IR systems increases. Searching (query processing) and indexing (creating data structures to enable fast searches) take more time and resources as the collection size grows. Larger collections lead to longer response times, which can make the system unusable.

Parallel IR uses multiple processors to speed up tasks, while Distributed IR spreads data across computers. These approaches improve speed and scalability for modern search systems like web search engines.

1.1 Parallel Computing

Parallel computing is the simultaneous application of multiple processors to solve a single problem, where each processor works on a different part of the problem.

  • Overall time = time required by the longest running subtask.
  • More processors → smaller parts → faster solution → scalable to larger problems.
Flynn's Taxonomy of Parallel Architectures
ClassFull NameDescriptionExample
SISD Single Instruction, Single Data Traditional sequential (von Neumann) computer Uniprocessor PCs
SIMD Single Instruction, Multiple Data N processors execute same instruction on N data streams simultaneously CM-2 (Thinking Machines)
MISD Multiple Instruction, Single Data N processors each execute different instructions on one shared data stream; rare Systolic Arrays
MIMD Multiple Instruction, Multiple Data N processors, N instruction streams, N data streams; most versatile & popular IBM SP2, SMP servers
🔑 Key Concept — MIMD MIMD is the most general class. Processors can work on separate tasks or cooperate on one task. MIMD systems with high processor interaction are tightly coupled; low interaction = loosely coupled. Distributed systems are also MIMD — just with higher inter-processor communication cost.

⚡ Granularity

  • Coarse-grained: much computation, little communication (distributed)
  • Fine-grained: much communication, little computation (shared memory MIMD)

💡 Coupling

  • Tightly coupled: high processor interaction, fast communication
  • Loosely coupled: low interaction, network-based (e.g., distributed IR)
1.2 Performance Measures
⚡ Speedup
S = Tseq / Tpar
T_seq = best sequential algorithm time  |  T_par = parallel algorithm time on N processors
Ideal: S = N (perfect speedup). In practice, unattainable.
📐 Amdahl's Law
S = 1 / ( J + (1 − J) / N )
J = fraction of problem that must be sequential (cannot be parallelised)
N = number of processors
As N → ∞, maximum speedup → 1/J
📊 Efficiency
E = S / N
S = speedup  |  N = number of processors
Ideal: E = 1 (no processor ever idle). In practice, unattainable.
⚠️ Why Perfect Speedup Is Unattainable Problem cannot always be divided into N equal subtasks · Parallel architecture imposes scheduling/synchronisation overhead · Some problems contain an inherently sequential component.
2

Parallel IR

2.1 Introduction

Parallel IR can be developed in two ways:

  • Develop new retrieval strategies that directly lend themselves to parallel implementation.
  • Adapt existing, well-studied algorithms to parallel processing.

Both MIMD and SIMD architectures can be used for IR tasks.

2.2 MIMD Architectures

🔀 Multitasking (Non-Cooperative)

  • Each processor runs an independent search engine.
  • A broker distributes incoming queries to processors.
  • Search engines do not share computation for a single query.
  • Shared resources: code libraries, file system caches, shared memory.
✅ Simple to implement, scales easily (more processors = more queries handled)
❌ Individual query response time unchanged; possible I/O bottlenecks

🤝 Cooperative Parallelism (Query-Level)

  • Multiple processors collaborate on one query.
  • Broker splits query → sends subtasks → processors evaluate parts → results merged.
  • Significantly improves query response time.
✅ Dramatically reduces latency for a single query
❌ More complex coordination between processors
Data Partitioning Strategies

📄 Document Partitioning

  • Collection is horizontally partitioned.
  • Each processor gets a subset of documents.
  • Every processor runs the full query on its subset.
  • Broker merges partial ranked lists.

🔤 Term Partitioning

  • Inverted index is vertically partitioned by terms.
  • Each processor handles a subset of index terms.
  • Query broken into terms → sent to responsible processor.
  • Broker aggregates partial document scores.
⚖️ When to Use Which? Skewed term distribution → Document partitioning performs better.  Uniform term usage in queries → Term partitioning performs better.
Inverted File Indexing in Parallel IR

🔗 Logical Document Partitioning

  • Global inverted file is shared.
  • Dictionary entries include P pointers (one per processor).
  • All processors read shared data without lock contention.
  • Index construction: each processor indexes a subset, lists merged via heap-based priority queue.

🗂️ Physical Document Partitioning

  • Each processor builds a self-contained index for its own documents.
  • No shared index during query processing.
  • Broker merges local hit-lists using priority queue.
  • Global term stats needed for consistent ranking (2-phase evaluation).
Other Index Structures
  • Suffix Arrays (Document Partitioning): Each processor builds a local suffix array; broker merges results.
  • Suffix Arrays (Term Partitioning): Array divided lexicographically; requires shared memory access — not well-suited to distributed environments.
  • Signature Files (Document Partitioning): Each processor builds signatures for its documents; query signature broadcast; local matching done in parallel.
  • Signature Files (Term Partitioning): Uses bit-sliced signature files; limited speedup and high false-positive removal cost — not recommended.
  • Threads (Java / POSIX) are used for implementing parallel processes, providing communication, concurrency, and synchronisation abstractions.
2.3 SIMD Architectures

SIMD architectures execute the same operation on multiple data points simultaneously. A key example is the Thinking Machines CM-2, applied to both signature-file and inverted-file IR algorithms.

  • Processing Element (PE): Each PE has a 1-bit ALU and local memory.
  • Local Parallel Instructions: Perform operations in parallel on each PE's local data.
  • Non-local Instructions: Inter-PE communication (e.g., finding a global maximum or vector sum).
  • Virtual Processors (VP): CM-2 abstracts physical processors into VPs; as VP ratio increases, execution time increases nearly linearly.
Signature Files on SIMD

Signature files are a natural fit for SIMD due to their binary, parallel nature.

  • probe.doc: Core subroutine — tests if a document signature matches a query term using hash functions; each VP handles a separate document signature simultaneously.
  • Segmented Signatures: If a document has more terms than fit in one signature, it is split into segments; a match in any segment counts.
  • Boolean Retrieval: Recursive structure (bquery.t) combining terms with AND, OR, NOT operators.
  • Ranked Retrieval: Query terms have weights; each match increases a document's score stored in a parallel variable (Doc.score).
  • Ranking Techniques: Global ranking (sorts all scores) vs. Iterative global max (finds top-k efficiently).
Optimising Signature File Access

📦 Batch Processing

  • If the full signature file doesn't fit in memory, processed in batches.
  • Multiple queries per batch amortise I/O costs.
  • Improves throughput but not response time.

🔲 Bit-Sliced Signature Files

  • Stored column-wise (bit-slices) instead of row-wise.
  • Only relevant bit-slices for the query are loaded.
  • Each VP handles one document's bits across slices.
Inverted Files on SIMD
  • Postings Table: Stores document IDs for each term; Index maps each term to a range in the postings table.
  • Scoring Algorithm: For each query term, look up postings → activate PEs → update document scores (accumulators).
  • Partitioned Postings File: Each document's postings and accumulator reside on the same processor → reduces communication overhead; supports sub-2-second query times on terabyte-scale data (CM-2 with 64K processors).
  • Space Overhead: ~33% (10–20% from padding in partitioned segments).

📌 SIMD Summary

  • Excellent for signature-based retrieval (especially when false positives are acceptable).
  • Inverted file search adapted using careful data layout and processor-aware partitioning.
  • Less flexible than MIMD but offers high-throughput parallel processing on large datasets.

Distributed IR

3.1 Introduction

Distributed computing applies multiple computers connected by a network to solve a single problem. Each node may be independently owned, can itself be a parallel computer, and communicates via TCP/IP (slower than shared memory in MIMD).

Distributed IR Architecture Components

🤝 Broker Process

  • Accepts client search requests.
  • Forwards requests to selected servers.
  • Collects intermediate results.
  • Combines them into a final ranked list.

🖥️ Server Processes

  • Each server resides on a separate node.
  • Processes document subsets or entire collections.
  • Communicates with the broker over the network.
  • Supports coarse-grained parallelism.
💡 Why Document Partitioning Suits Distributed IR Documents can be divided into collections fitting naturally on separate servers · Each server processes its self-contained subcollection (reduces inter-server communication) · Workload is computationally intensive but communication-light.
⚠️ Why Term Partitioning Is Not Ideal for Distributed IR Requires significant inter-server communication for merging partial results · Increases response latency due to synchronisation overhead.
Engineering Issues
  • Search Protocol: Defines message syntax, sequence for server discovery, query submission, result retrieval; uses TCP/IP as transport.
  • Server Design: Must accept/parse requests, spawn threads per request, apply caching, optimise for locality of reference.
  • Broker Design: Accepts queries asynchronously, selects relevant servers, distributes query in parallel, merges results into a final ranked list.
Standard Search Protocols

📚 Z39.50

  • Widely adopted client/server IR protocol.
  • Supports search, retrieval, database discovery.
  • Suitable for library systems and academic databases.

🌐 STARTS

  • Stanford Proposal for Internet Meta-Searching.
  • Addresses merging of results from different sources.
  • Standardises scoring and relevance across systems.
3.2 Collection Partitioning

Collection partitioning refers to how documents are assigned across different search servers. The strategy depends on administration, collection size, and search goals.

1. Independent / Heterogeneous Systems
  • Documents independently indexed and managed at each server — no central control.
  • Each server often focuses on a specific topic → semantic partitioning (legal, medical, tech).
  • Common in meta-search systems (federated search engines).
2. Centrally Administered Systems

🔁 (a) Replicated Partitioning

  • Entire collection copied to every server.
  • Used for high availability and load balancing.
  • ⚠️ Updates/deletions must be broadcast to all servers.

🎲 (b) Random Distribution

  • Documents distributed randomly among servers.
  • Every query broadcast to all servers; results merged.
  • Best when users expect to search the entire collection.

🏷️ (c) Semantic Partitioning

  • Documents organised by topic/category (sports, medicine, finance).
  • Done manually or using automatic clustering/categorisation techniques.
  • Improves search precision by sending queries to topic-relevant servers.
3.3 Source Selection

Source selection is the process of determining which distributed document collections are most likely to contain relevant documents for the current query.

Approaches to Source Selection
  • 1. Broadcast to All Servers: Simple but inefficient. Used when distribution is random, collections overlap semantically, or communication costs are low.
  • 2. Ranked Collection Selection: Each collection is treated as a "super-document"; query is compared against collection summaries using cosine similarity and tf-idf weighting.
    Problem: False Drops — a collection may rank highly overall but no individual document inside may actually be relevant. Solution (Moffat & Zobel): partition each collection into blocks (e.g., every 10 docs) and index blocks instead of whole collections.
  • 3. Content Modeling (Voorhees): Use training queries to build a model for each collection's content; when a new query arrives, match it to similar training queries and predict which collections to query and how many results to expect.
📐 tf-idf for Collection Ranking
idf_i = log(N / n_i)
tf_ij = frequency of term i in collection j  |  N = number of collections  |  n_i = number of collections containing term i
3.4 Query Processing

Query processing in distributed IR is a multi-step procedure where user queries are handled by multiple distributed servers to retrieve relevant documents efficiently.

Steps in Query Processing
  1. Select Collections to Search — Choose relevant collections based on query content (source selection or semantic matching).
  2. Distribute the Query — Broker sends the query to selected collections using a defined search protocol (e.g., TCP/IP, STARTS, Z39.50).
  3. Evaluate Query in Parallel — Each search server evaluates the query independently using its local index and algorithm.
  4. Merge Results — Broker collects and merges partial results into a final ranked list to return to the user.
Result Merging Strategies
  • Boolean Queries: Servers return Boolean result sets; final result = union of all sets.
  • (a) Round-Robin Interleaving: Simple interleaving of top results from each server. ❌ Gives equal weight to all collections regardless of relevance.
  • (b) Score-Based Merging: Combine documents by relevance score. ✅ Works well when term statistics are globally consistent. ❌ Can fail with semantic partitions or independent servers.
  • (c) Reranking with Collection Weights (Callan): Adjust scores based on collection relevance.
📐 Collection Weight Formula
w_i = 1 + |C| × (s̄ − s_i) / s̄
|C| = number of collections  |  s_i = score of collection i  |  s̄ = average collection score
Prioritises results from more relevant collections.
  • 3. Global Term Statistics-Based Merging: Most accurate but complex. Two methods: (i) use existing global index included with query; (ii) two-round communication — Round 1: servers return collection statistics; Round 2: broker computes global stats and resends for improved scoring.
  • 4. Broker-Side Reranking: Servers return term statistics per document; broker reranks all documents centrally. Produces results as if all documents were in a single collection.
3.5 Web Issues

🏢 Centralised Web Search (Current Model)

  • All crawled web documents stored and indexed centrally.
  • Distributed and parallel IR used internally for scalability.
  • Examples: Google, Bing.

🕸️ Truly Distributed Web IR (Harvest System)

  • Web's distributed nature is preserved.
  • Gatherers: collect content from various sources.
  • Brokers: handle queries, manage indexing and results.
  • Broker Registry: helps users find the right broker for their topic.

Searching the Web

4.1 Introduction

The World Wide Web originated in the late 1980s and can be seen as a very large, unstructured but ubiquitous database — estimated at over one terabyte of textual data, plus images, audio, and video. This triggers the need for efficient tools to manage, retrieve, and filter information.

We focus on syntactic search — searching for user-specified words or patterns in text. Natural language analysis is an alternative but is too costly and not yet very effective at large scale.

Three Forms of Searching the Web
  • Search Engines: Index a portion of web documents as a full-text database.
  • Web Directories: Classify selected web documents by subject (human-curated).
  • Hyperlink-Based Search: Exploits the hyperlink structure of the web (not yet fully available).
4.2 Challenges
Data-Related Problems
  • Distributed Data: Data spans many computers and platforms with varying bandwidth and reliability.
  • High Volatility: ~40% of the web changes every month; dangling links and relocation problems are common.
  • Large Volume: Exponential growth poses difficult scaling issues.
  • Unstructured & Redundant Data: ~30% of web pages are near-duplicates; the web uses "semi-structured" data, unlike a true hypertext with a conceptual model.
  • Poor Data Quality: No editorial process; data can be false, invalid, or poorly written. Typos range from 1 in 200 (common words) to 1 in 3 (foreign surnames).
  • Heterogeneous Data: Multiple media types, formats, languages, and alphabets (e.g., Chinese Kanji, Japanese).
User-Related Problems
  • How to specify a query: Users often don't understand the system's logical view of text (case sensitivity, stop words, stemming, Boolean logic).
  • How to interpret the answer: Results can be too large, poorly ranked, or not up to date.
4.3 Search Engines

Search engines are specialised IR systems designed for massive-scale web data. They rely entirely on precomputed indexes because accessing the full text of documents at query time is impractical (cannot store local copies of the entire web; real-time retrieval is too slow).

4.3.1 Centralised Architecture

Most search engines use a centralised crawler-indexer architecture. Crawlers (also called robots, spiders, wanderers, or walkers) traverse the web, sending new or updated pages to a main server to be indexed. The index is then used to answer queries.

  • Main problems: gathering data from the highly dynamic web, saturated communication links, high server load, and volume of data.
  • Good load balancing between crawling, indexing, and query answering is critical.

4.3.2 Distributed Architecture (Harvest)

The Harvest system uses a distributed architecture to gather and distribute data — more efficient than the centralised crawler architecture.

  • Gatherer: Collects and extracts indexing information from one or more web servers. Runs on a web server → generates no external traffic for that server.
  • Broker: Provides the indexing mechanism and query interface to the gathered data; updates its index incrementally by retrieving from gatherers or other brokers.
  • Broker Registry: Allows brokers to register and users to find the right broker for their topic.
  • Replicators: Replicate servers for scalability (e.g., registration broker replicated in different geographic regions).
  • Object Cache: Reduces network/server load and response latency.
4.4 User Interfaces
  • Query Interface: Basic = one or more words. Advanced = Boolean operators, phrase search, proximity search, wildcards, field-specific filters (URL, title, language, date range, file type).
  • Answer Interface: Typically shows top 10 ranked pages. Each entry includes URL, size, indexing date, and a snippet (title + first lines or selected headings).
  • Inconsistencies: Different engines handle stop words, stemming, and case sensitivity differently — users may not be aware.
  • Refinement: Users can refine queries using complex Boolean expressions or find similar pages.
4.5 Ranking

Most search engines use variations of the Boolean or vector model for ranking. Some also consider metatag terms, page titles, or page popularity.

🔗 WebQuery

  • Ranks pages based on how connected each page is.
  • Extends the result set by finding highly connected related pages.
  • Allows visual browsing of web pages.

🎯 HITS (Kleinberg)

  • Query-dependent ranking.
  • Authorities: pages with many incoming links (relevant content).
  • Hubs: pages with many outgoing links (point to relevant content).
  • Positive two-way feedback between hubs and authorities.
📐 HITS — Hub & Authority Equations
H(p) = Σ A(q)   for all q that p points to
A(p) = Σ H(q)   for all q that point to p
Values converge to the principal eigenvector of the link matrix of S.
S = set of pages that point to or are pointed to by pages in the query result.
📐 PageRank (Google) — Markov Chain Model
PR(a) = q + (1 − q) × Σ PR(p_i) / C(p_i)
q = probability of jumping to a random page (typical value: 0.15)  |  1−q = probability of following a random hyperlink
C(e) = number of outgoing links of page e  |  p_i = pages pointing to page a
PageRank simulates a random web surfer and computes the stationary probability of being on each page.
⚠️ HITS Limitations Does not work with non-existent, repeated, or automatically generated links · Topic of result can become diffused · Avoided by weighting links based on surrounding content.
4.6 Crawling the Web

Crawling is the process of discovering and downloading web content. Crawlers start from a set of seed URLs and extract more URLs recursively.

🌊 Breadth-First Traversal

  • Wide and shallow — fetches all immediate links before going deeper.
  • Good for broad coverage.

🔽 Depth-First Traversal

  • Deep and narrow — follows links down to max depth before backtracking.
  • Good for focused topics.
Crawl Management
  • Avoid fetching the same page multiple times.
  • Respect bandwidth and server limits.
  • Handle page updates efficiently.
  • Assign crawlers by domain or geographic region.
  • Maintain a crawl history; use robots.txt to honour crawl restrictions.

🕐 Crawl Freshness

  • Crawled pages can be 1 day to 2 months old.
  • Popular/frequently updated pages crawled more often.
  • Some crawlers learn update frequencies to optimise revisits.

⚙️ Technical Issues

  • Robots must avoid overwhelming servers.
  • Pages with frames or authentication are hard to crawl.
  • Broken link percentage in search engine indices: 2–9%.
4.7 Indices

An index is a data structure enabling fast retrieval of documents matching user queries. The main technique is the inverted file index.

  • Inverted File Index: Contains a sorted vocabulary (unique words) and for each word, a posting list of documents where it appears.
  • Compression: Stop word removal, stemming, normalization + compressed posting lists. Index size ≈ 30% of original text (down to 10% with compression).
  • Scale: For 100 million pages, approximately 150 GB of index storage is needed.
  • Additional Metadata: URL, document title, snippets (first lines/headings), creation date and size.
  • Query Processing: Answered by binary search on the vocabulary; multi-word queries require merging of posting lists.
  • Phrase/Proximity Queries: Require storing word positions; full position indexing is space-consuming.
  • Wildcards: Prefix queries use two binary searches; complex patterns use sequential scan over the vocabulary.
  • Block-based Indexing: Reduces index size by indexing logical blocks instead of full documents (used in Glimpse and Harvest).
4.8 Browsing

Web tools based on browsing — particularly Web Directories — offer much more relevant results, even though their coverage is very low (less than 1% of all web pages).

4.9 Web Directories

Web directories (also called catalogues, yellow pages, or subject directories) are hierarchical taxonomies that classify human knowledge. The best-known example is Yahoo!. Others include eBLAST, LookSmart, Magellan, NewHoo.

  • Pages must be submitted to the directory; reviewed and, if accepted, classified in one or more categories.
  • The taxonomy is technically a directed acyclic graph (cross-references exist).
  • Yahoo! has close to one million pages classified; 14 regional/language-specific directories.
  • Classification is done manually — automated natural language processing is not yet 100% effective.

✅ Advantages

  • If you find what you're looking for, the answer is almost always useful.
  • Much more relevant results than general search engines.
  • Human-reviewed quality.

❌ Disadvantages

  • Classification is not specialised enough.
  • Not all web pages are classified.
  • Coverage gets worse every day as the web grows.
  • Users may have different notions of categories than the curators.
4.10 Combining Searching with Browsing

WebGlimpse is a tool that combines browsing with searching. It attaches a small search box to the bottom of every HTML page, allowing the search to cover the neighbourhood of that page or the whole site — without having to stop browsing.

  • Neighbourhood: Set of web pages reachable by a path of hypertext links within a maximum predefined distance (configurable separately for local and remote pages).
  • WebGlimpse indexes a web site and computes neighbourhoods according to user specifications, adds search boxes to selected pages, collects and caches relevant remote pages.
  • Uses Glimpse as its underlying search engine.
  • Useful for building indices for personal web pages or collections of favourite URLs.
4.11 Metasearchers

Metasearchers are web servers (or clients) that send a given query to several search engines, web directories, and other databases, collect the answers, and unify them. Examples: Metacrawler, SavvySearch, Copernic, EchoSearch, WebFerret.

  • Allow the user to pose the same query to multiple sources through a single interface.
  • Results can be sorted by different attributes (host, keyword, date) — often more informative than a single engine's output.
  • Pages returned by more than one search engine should be more relevant.
  • Limitation: Results are not necessarily all matching pages; the number of results retrieved per engine is limited.
💡 Advanced: Inquirus (NEC Research Institute) Downloads and analyses each web page obtained, displays each page with query terms highlighted. Results displayed progressively as they are available. Non-existent or changed pages are discarded. Provides better ranking — but not available to the general public.
4.12 Finding the Needle in the Haystack — User Problems
  • Users do not always understand the logical view of text adopted by the system (e.g., case sensitivity, stop words).
  • Typos / variants: 10–20% of matches can be lost; foreign names up to 50% loss.
  • Boolean logic misuse: ~80% of queries use no Boolean operations; natural language "and/or" differs from Boolean interpretation.
  • Query behaviour: 25% use only 1 keyword; average 2–3 terms; ~15% restrict by predefined topic; ~80% don't modify the query; ~85% only view the first screen of results; 64% of queries are unique.
Teaching the User — Tips for Better Searching
  • Give as many query terms as possible, including terms that must or must not appear.
  • Include all possible synonyms of important words.
  • Restrict search to a field (title, URL) or limit attributes (date, country) to reduce result size.
  • Consult the search engine's help documentation.
  • Use the browser's "Find" option to locate a term within a retrieved page.
📌 Rules of Thumb Specific queries → Look in an encyclopaedia / library.   Broad queries → Use Web directories for good starting points.   Vague queries → Use Web search engines and iteratively improve the query based on relevant answers.
4.13 Searching Using Hyperlinks
Web Query Languages

Queries can include the link structure connecting web pages, not just content. For example: "find all pages containing at least one image reachable from a given site following at most three links."

  • First Generation: Combine content patterns with graph queries describing link structure (path regular expressions). Includes W3QL, WebSQL, WebLog, WQL.
  • Second Generation (Web Data Manipulation Languages): Extend to include internal page structure and allow creation of new structures as query results. Includes STRUQL, FLORID, WebOQL.
Dynamic Search and Software Agents

Dynamic search follows links online to discover relevant information — searching the current structure of the web, not the precomputed index. Useful for small, dynamic subsets of the web.

  • Fish Search: Exploits the intuition that relevant documents often have relevant neighbours; guides search by following links in relevant documents.
  • Shark Search: Improves Fish Search with a better relevance assessment of neighbouring pages. Both algorithms follow links in priority order from a single starting page, using a heuristic to decide whether to follow links on relevant pages.
  • Software Agents: Search specific information across heterogeneous sources. Key issues: determining relevant sources and merging results (the fusion problem). Examples: Jango, Junglee, Express (shopping robots).

📌 Unit IV — Key Takeaways

  • Parallel IR uses MIMD or SIMD architectures; key strategies are document partitioning and term partitioning.
  • Amdahl's Law limits speedup: maximum = 1/J where J is the sequential fraction.
  • Distributed IR uses a broker-server model; document partitioning suits it best due to coarse granularity.
  • Source selection (broadcast, ranked collection, content modeling) is crucial for efficiency.
  • Result merging strategies range from simple round-robin to global term statistics-based reranking.
  • Web search engines rely on centralised crawler-indexer architectures; Harvest offers a distributed alternative.
  • PageRank (Google) simulates a random surfer; HITS uses hub-authority feedback — both exploit hyperlink structure.
  • Web directories offer high relevance but low coverage; search engines offer broad coverage but lower precision.
  • Metasearchers and WebGlimpse bridge gaps between searching and browsing.
